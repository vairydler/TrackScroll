## IDタグについて

各仕様にはユニークなIDタグ `[SPEC-XXX]` を付与する。
コード内では以下の形式で参照する。

```python
# [SPEC-TRIGGER-COUNT-RESET] トリガ変更時にカウントをリセット
self._reset_count()
```

## 1. 用語定義

| 用語 | 定義 |
|------|------|
| **ジェスチャテーブル** | ジェスチャレコードが複数登録されたテーブル |
| **ジェスチャレコード** | トリガと、複数のジェスチャデータの組み合わせ |
| **ジェスチャデータ** | ジェスチャキー・ダウン動作・アップ動作・リピート動作・リピート開始時間の組み合わせ |
| **トリガ** | フックを開始するマウスボタンおよびキーの組み合わせ |
| **ジェスチャキー** | ジェスチャ動作を起動するためのキー。マウスの移動方向・マウスボタン・スクロール・キーのいずれか |
| **ダウン動作** | ジェスチャキーのダウン発行条件が成立した時点で発火する動作 |
| **アップ動作** | ジェスチャキーが離された時点（またはトリガ変更時）に発火する動作 |
| **リピート動作** | リピート開始時間経過後、リピート間隔ごとに繰り返し発火する動作 |
| **リピート開始時間** | ジェスチャキー入力後、何も操作しない場合にリピートを開始するまでの待機時間 |
| **猶予時間** | トリガ押下からダウン動作を発行するまでの待機時間。この時間以内にトリガを離した場合、または他のジェスチャが実行された場合は何も起きない |
| **移動クールダウン時間** | 同一方向へのジェスチャ動作が実行された後、同じ方向への再入力を無視する時間 |
| **トリガ変更** | アクティブなトリガが変化すること（押し増し・押し減らし・トリガなしへの移行を含む） |
| **フック** | マウス移動およびキー入力をジェスチャ目的でキャプチャしている状態 |

## 2. ジェスチャテーブル

### 2.1 構造

```
ジェスチャテーブル
└── ジェスチャレコード（複数）
    ├── トリガ（キーとマウスボタンの組み合わせ）
    ├── 移動方向判定しきい値（省略時はシステムデフォルト値を使用）
    ├── 移動クールダウン時間（省略時は 0.0 = 無効）
    └── ジェスチャデータ（複数）
        ├── ジェスチャキー
        ├── ダウン動作（省略可）
        ├── アップ動作（省略可）
        ├── リピート動作（省略可）
        └── リピート開始時間
```

### 2.2 設定ファイル

`[SPEC-CONFIG-FILE]`
ジェスチャテーブルおよびシステムパラメータは、単一の設定ファイルから取得する。
現時点ではPythonコード上に直接記述し、特定の関数の戻り値として返す形式とする。
将来的にはJSON設定ファイルからの読み込みへ移行できるよう、各取得関数のインターフェースを維持すること。

設定ファイルが提供する取得関数：

| 取得関数 | 内容 |
|------|------|
| ジェスチャテーブル取得 | ジェスチャレコードの一覧 |
| 移動方向判定しきい値取得（デフォルト） | レコード未指定時に使用するしきい値距離 |
| 手ブレ補正しきい値取得 | リピートタイマリセットに用いる移動量 |
| リピート間隔取得 | リピート発火間隔（全ジェスチャ共通） |
| 猶予時間取得 | トリガ自体へのダウン動作発行を遅延させる時間 |

### 2.3 競合チェック

`[SPEC-CONFLICT-DETECT]`
以下の競合が存在する場合、ジェスチャテーブルのロード時に検出し、GUIポップアップで通知する。

- あるトリガに登録されたジェスチャキーCと、そのトリガにCを加えた組み合わせ（トリガ+C）が別のジェスチャレコードのトリガとして登録されている場合。

`[SPEC-CONFLICT-PRIORITY]`
この競合は運用上の設定ミスとして扱い、システムはトリガを優先して動作する。

`[SPEC-INVALID-UP-DETECT]`
移動方向・スクロール系のジェスチャキーにアップ動作が登録されている場合、ロード時に検出しGUIポップアップで警告する。これらのジェスチャキーはアップ動作を持たない（「離す」という概念がないため）。

## 3. トリガ

### 3.1 構成

`[SPEC-TRIGGER-COMPOSITION]`
トリガはキーとマウスボタンの組み合わせのみとする。マウス移動・スクロールなど、入力状態を維持できないものはトリガに使用できない。

### 3.2 トリガの監視

`[SPEC-TRIGGER-ALWAYS-WATCH]`
トリガはフック状態によらず常時監視する。

### 3.3 トリガ変更

`[SPEC-TRIGGER-CHANGE]`
アクティブなトリガは「現在押されているボタン・キーの組み合わせ全体」で決まる。アクティブなトリガが変化する以下のケースをトリガ変更と呼ぶ。

- **押し増し**：`A` → `A+B`
- **押し減らし**：`A+B` → `A`
- **フック開始**：トリガなし → `A`
- **フック終了**：`A` → トリガなし

`[SPEC-TRIGGER-CHANGE-SWITCH]`
トリガ変更が発生した場合、旧トリガのフックは終了し、新トリガのフックを開始する。

`[SPEC-TRIGGER-CHANGE-NO-ROLLBACK]`
旧フック中に実行済みのジェスチャ動作はロールバックしない。

`[SPEC-TRIGGER-COUNT-RESET]`
トリガ変更のたびに、新しいトリガのリピート開始時間および猶予時間のカウントをゼロからやり直す。

## 4. フック動作

### 4.1 フック中の入力制御

`[SPEC-HOOK-BLOCK-CURSOR]`
フック中かつ現在アクティブなジェスチャレコードに移動方向ジェスチャキー（MOVE_UP / MOVE_DOWN / MOVE_LEFT / MOVE_RIGHT）が1つ以上登録されている場合、マウスカーソルの移動を禁止する。マウスの移動量は参照するが、実際のカーソル位置には反映させない。
移動方向ジェスチャキーが1つも登録されていない場合は、フック中であってもカーソル移動を禁止しない。

`[SPEC-HOOK-BLOCK-GESTURE-KEY]`
フック中は現在のトリガレコードに登録されているジェスチャキーの入力伝播を停止する。ジェスチャ動作の成否にかかわらず、フック中は常時停止する。

`[SPEC-HOOK-NO-MATCH-PASSTHROUGH]`
フック中に入力されたジェスチャキーが、現在のトリガレコードのいずれのジェスチャデータにも該当しない場合は、その入力を伝播する。

`[SPEC-HOOK-BLOCK-TRIGGER-KEY]`
トリガに割り当てられたキー・ボタンは、フック状態によらず常時、本来の動作の伝播を停止する。
猶予時間以内にトリガを離した場合（何も起きない）も含め、完全に抑制する。

`[SPEC-SELF-EVENT-FILTER]`
本ソフトが送出したイベントは、フックコールバックで無視する。
送出イベントの `dwExtraInfo` に識別子を付与し、フック受信時に照合することで判定する。
実装の詳細は10章を参照。

### 4.2 フック外の動作

`[SPEC-HOOK-OUT-PASSTHROUGH]`
フックしていない状態では通常の入力動作を一切妨げない。ただし `[SPEC-HOOK-BLOCK-TRIGGER-KEY]` は例外とする。

### 4.3 フック状態の解除保証

`[SPEC-HOOK-RELEASE-GUARANTEE]`
何の物理ボタンも押していない状態でフックが継続することは絶対に避けなければならない。

## 5. ジェスチャキー

### 5.1 種類

| 種類 | 説明 | ダウン | アップ |
|------|------|--------|--------|
| マウス移動方向 | 上・下・左・右の4方向 | ○（方向確定時） | × |
| マウスボタン | 左・右・中・X1・X2など | ○ | ○ |
| スクロール | 垂直（上/下）、水平（左/右） | ○（イベント受信時） | × |
| キー | 任意のキーボードキー | ○ | ○ |
| トリガ自体（空欄） | 空欄ジェスチャキー | ○（フリック時・リピート開始時） | ○（フック終了時） |

`[SPEC-GKEY-SINGLE]`
ジェスチャキーは1つのジェスチャデータにつき1つの入力のみ登録する。

### 5.2 マウス移動方向の判定

`[SPEC-MOVE-THRESHOLD]`
累積移動量がしきい値距離を超えた時点で、累積移動ベクトルの角度から上下左右の方向を確定する。
しきい値距離はジェスチャレコードごとに設定できる。省略時はシステムデフォルト値を使用する。

`[SPEC-MOVE-RESET]`
ジェスチャ動作が実行されるたびに累積移動量をリセットする。

`[SPEC-MOVE-COOLDOWN]`
ジェスチャレコードに移動クールダウン時間（`move_cooldown`）を設定できる。省略時または `0.0` の場合は無効（従来通り）。

クールダウンが有効な場合、方向が確定するたびに「最後に確定した方向」と「その時刻」を記録する（タイマーは1つ）。
次の方向確定時に以下のように判定する。

| 条件 | 動作 |
|------|------|
| 直前と**同一方向** かつ 経過時間が `move_cooldown` 未満 | ジェスチャ動作を実行しない。累積移動量はリセットする |
| 直前と**別方向** | タイマーをリセットしてジェスチャ動作を通常通り実行 |
| 直前と**同一方向** かつ 経過時間が `move_cooldown` 以上 | クールダウン終了としてジェスチャ動作を通常通り実行 |

別方向の確定がタイマーをリセットするため、`↑←↑` は受け付けるが `↑↑` は抑制される（クールダウン時間内の場合）。
別方向のジェスチャキーがレコードに登録されているかどうかは判定に影響しない。

クールダウンが抑制するのはダウン動作の発火のみ。リピート動作が定義されている場合、リピートはクールダウンによる抑制を受けない。

**クールダウン対象：**
移動方向ジェスチャキー（上・下・左・右）のみ。スクロール・ボタン・キー種別のジェスチャキーには適用しない。

**トリガ変更時の扱い：**
クールダウン状態（最終確定方向・時刻）はセッションに属する。トリガ変更でセッションが破棄されると、クールダウン状態もリセットされる。

| move_cooldown 値 | 動作 |
|------|------|
| `None` または `0.0` | クールダウン無効（従来通り） |
| `0.0` 超の値 | 同一方向連続入力を指定秒間抑制 |

### 5.3 空欄のジェスチャキー（トリガ自体への動作割り当て）

`[SPEC-TRIGGER-ACTION-ASSIGN]`
ジェスチャキーを空欄で登録した場合、そのジェスチャデータはトリガ自体への動作割り当てとなる。

`[SPEC-TRIGGER-ACTION-DELAY]`
トリガ自体のダウン動作は、以下の2条件のいずれかが成立した時点で発行する。

- **フリック**：猶予時間以内にトリガを離した場合 → ダウン発行後、即アップ発行
- **リピート開始**：猶予時間経過 かつ リピート開始時間経過 → ダウン発行後、リピート開始

以下の場合はダウン動作を発行しない（アップ動作も発行しない）：

- 猶予時間以内に他のジェスチャキーが実行された場合（猶予時間カウントを停止）
- 猶予時間が経過したがフリックもリピート開始も成立しない場合

`[SPEC-TRIGGER-ACTION-PUSHONLY]`
トリガ自体へのダウン動作は、トリガが**押し増しによって新たにアクティブになった場合のみ**発行対象とする。
押し減らしによって復帰したトリガはダウン発行の対象としない。

| シナリオ | ダウン発行対象か |
|------|------|
| なし → A（Aを新たに押した） | ○ |
| A → A+B（Bを押し増した） | ○ |
| A+B → A（Bを離して戻った） | × |
| A+B → B（Aを離して戻った） | × |

`[SPEC-TRIGGER-ACTION-FLICK-TIME]`
猶予時間は設定ファイルから取得する。リピート開始時間と独立したパラメータ。
猶予時間とリピート開始時間のカウントは並列で進む。

**猶予時間とリピートの関係：**
- 猶予時間 = リピート開始時間の場合：リピート開始と同時にダウン発行
- リピート開始時間 < 猶予時間の場合：猶予時間経過でリピート開始時間も満たしているため即ダウン発行・リピート開始

**連打シナリオ（A・B・A+B それぞれにトリガ割当がある場合）：**

1. A押下 → `A` トリガ開始、猶予時間カウント開始
2. B押下 → `A+B` トリガへ変更、カウントリセット
3. B離上（A+Bの猶予時間以内）→ ダウン未発行→アップなし、`A` トリガへ戻り
   → `[SPEC-TRIGGER-ACTION-PUSHONLY]` により押し減らしのためダウン発行対象外
4. 2〜3の繰り返し中、Aのダウンは発行されない
5. 連打停止後、A押下時点からの猶予時間でフリック or リピート開始が成立した場合のみダウン発行

## 6. ジェスチャ動作

### 6.1 登録できる動作の種類

| 種類 | 説明 |
|------|------|
| キー入力 | 仮想キーコードによるキー操作（press / down / up） |
| マウスボタン | クリック・押下・離上など |
| スクロール | 垂直・水平スクロール |
| 関数 | 任意のPython関数 |

### 6.2 ダウン/アップ/リピートの3動作方式

`[SPEC-ACTION-THREE-WAY]`
各ジェスチャデータにはダウン・アップ・リピートの3種類の動作を個別に登録できる。
すべて省略可能（None扱い）。フィールドの記述順序は以下の通り：

```python
# (ジェスチャキー, ダウン, アップ, リピート, repeat_start)
# ダウンのみ（アップ・リピート省略）
(GKey.MOVE_UP, "Tab", 0.3)
# ダウン＋アップ（リピート省略）
(VK.XBUTTON1, "↓Shift", "↑Shift", 0.0)
# ダウン＋アップ＋リピート
(GKey.MOVE_UP, "Tab", None, "Tab", 0.3)
```

`[SPEC-ACTION-DOWN-UP-PAIR]`
「ダウン発行条件を満たしたら必ずアップを発行する」を原則とする。
ダウン発行条件を満たしていない場合はアップも発行しない。この原則はトリガ変更・フック終了・stop()呼び出し時にも適用する。

`[SPEC-ACTION-DOWN-KEY-TRACKING]`
セッションはダウン発行済みのジェスチャキー一覧をスタックとして管理する。
セッション破棄時（トリガ変更・フック終了）にこのスタックを逆順（後入れ先出し）で走査してアップを発行する。

### 6.3 簡易文字列記法

`[SPEC-ACTION-STRING-NOTATION]`
マウスボタン・スクロール・キーを登録する場合は、AutoHotkeyの記法を参考にした文字列形式で簡易登録できる。

モディファイヤ記号：

| 記号 | 意味 |
|------|------|
| `!` | Alt |
| `^` | Ctrl |
| `#` | Win |
| `+` | Shift |

例：`!^#C` → Alt+Ctrl+Win+C

**モディファイヤ＋マウスボタン：**

モディファイヤ記号はマウスボタンにも付与できる。
発行順序はキー入力と同様に、モディファイヤ down → ボタン click → モディファイヤ up（逆順）となる。

| 記法 | 動作 |
|------|------|
| `^LButton` | Ctrl down → LButton click → Ctrl up |
| `!RButton` | Alt down → RButton click → Alt up |

**モディファイヤ＋ホイール：**

モディファイヤ記号はホイールにも付与できる。
発行順序はモディファイヤ down → スクロール送出 → モディファイヤ up（逆順）となる。

| 記法 | 動作 |
|------|------|
| `^WheelUp` | Ctrl down → WheelUp → Ctrl up |
| `+WheelDown` | Shift down → WheelDown → Shift up |

押しっぱなし記法：

| 記法 | 意味 |
|------|------|
| `↓Alt` | key_down(Alt) |
| `↑Alt` | key_up(Alt) |
| `↓+Ctrl` | key_down(Shift) + key_down(Ctrl) |
| `↑+Ctrl` | key_up(Ctrl) + key_up(Shift)（逆順） |
| `↓^LButton` | Ctrl down → LButton down |
| `↑^LButton` | LButton up → Ctrl up（逆順） |

`↓` / `↑` プレフィックスはモディファイヤ付きキー・マウスボタンに使用できる。
ホイールは押下状態を持たないため、`↓` / `↑` プレフィックスとの組み合わせはエラーとする。
プレフィックスの有無で key_press / key_down / key_up を切り替える。

### 6.4 アップ動作の発火条件

`[SPEC-ACTION-UP-TRIGGER]`
アップ動作は以下の2条件のいずれかで発火する。

- ジェスチャキーを離したとき
- トリガ変更があったとき

次の入力があっただけでは発火しない。
リピートが一度も発火しなかった場合もアップ動作は発火する。

`[SPEC-ACTION-UP-SUPPRESS]`
アップ動作を発行しない唯一の条件は「ダウン発行条件を満たしていない」こと。
ダウン発行条件とは、猶予時間経過（トリガ自体）またはジェスチャキー入力（その他）が成立したことを指す。
ダウン動作が `None` であっても、発行条件を満たした以上「発行済み」として扱う。
リピートの発火有無・経過時間・他ジェスチャの実行有無は抑制条件としない。

## 7. リピート動作

### 7.1 リピートタイマ

`[SPEC-REPEAT-START]`
リピートタイマのカウントは以下のタイミングで開始する。

- トリガがアクティブになったとき（フック開始・トリガ変更を含む）
- ジェスチャ動作が実行されたとき

リピート開始時間を超えた時点でリピートが発火する。ただしトリガ自体の動作の場合はダウン発行済みであることが条件。

`[SPEC-REPEAT-KIND]`
ジェスチャキーの種別によりリピート開始条件を分ける。

| 種別 | リピート開始条件 |
|------|----------------|
| キーボードキー | ダウンイベントから `repeat_start` 経過（押し続けている間） |
| マウスボタン | 同上 |
| マウス移動方向 | 最後に方向確定してから `repeat_start` 経過 |
| スクロール | 最後にスクロールイベントを受けてから `repeat_start` 経過 |

`[SPEC-REPEAT-TARGET]`
リピート対象はリピート動作のみ。ダウン動作はリピートしない。

`[SPEC-REPEAT-SINGLE]`
リピートは常に1つだけ走行する。複数同時走行しない。
新しいジェスチャ動作が実行されたタイミングでタイマリセット、リピート対象も新しいジェスチャに切り替わる。

`[SPEC-REPEAT-CLEAR-ON-MOVE]`
手ブレ補正しきい値（設定ファイルから取得）を超えるマウス移動があったとき、リピートタイマをリセットする。
全ジェスチャキー種別に適用する（カーソルが動き始めたらリピートをリセット）。

`[SPEC-REPEAT-CLEAR-ON-NEW-GESTURE]`
別のジェスチャ動作が実行されたとき、リピートタイマをリセットし対象を新しいジェスチャに切り替える。

`[SPEC-REPEAT-INTERVAL]`
リピート発火間隔はシステム全体で一律とする。設定ファイルから取得する。
リピート発火間隔の判定はメインループスレッドの周期で行う。別スレッドでリピートイベントをキューに積む方式は採用しない（リピート処理時間がリピート間隔を超えた場合にキューが際限なく積まれるため）。

### 7.2 リピート停止

`[SPEC-REPEAT-STOP]`
リピートは以下の条件で停止する。

- 別のジェスチャ動作が実行されたとき（対象の切り替え）
- トリガ変更があったとき
- リピート中のキーが離されたとき（アップ動作を発行してから停止）

### 7.3 リピート無効

`[SPEC-REPEAT-DISABLED]`
リピート動作が `None` の場合、リピートタイマ自体を無効にする最適化を許容する。
`repeat_start=0` はリピート動作が登録されていれば最速連射（ダウン発行後即リピート）として動作する。

---

## 8. ジェスチャキー同時押し

`[SPEC-GKEY-SIMULTANEOUS]`
ジェスチャキーの同時押しは定義しない。先着順に個別処理する。

- 先に押したキーのアップ動作は、後から押したキーのダウン時には発行しない。キーを離したときに発行する
- 複数のキーを同時に離した場合は、離した順番通りに処理する

---

## 9. 強制終了・異常系

`[SPEC-STOP-CLEANUP]`
`stop()` 呼び出し時、ダウン発行済みの全ジェスチャキーのアップ動作を逆順で発行してから終了する。
`[SPEC-ACTION-DOWN-UP-PAIR]` の原則をstop()にも適用する。

~~`[SPEC-TIMEOUT-RESET]`~~
~~キーボードキーのトリガが強制解除された場合（OS割り込み等）、キーリピートイベントの途絶をタイムアウト監視で検出し、セッションをリセットする（ベストエフォート）。~~

キーボードキーのトリガが強制解除された場合（OS割り込み等）の検出は行わない。
Shift等の押し増し時にOSが他キーのリピートを止める挙動により、キーリピートの途絶による検出アプローチは誤作動を招くため廃止した。
マウスボタンと同様に、強制解除時の対処はしない。

## 10. 実装上の制約・注意事項

- `[SPEC-HOOK-BLOCK-CURSOR]` の実現には `input_driver.py` の `on_mouse_move_filter`（伝播停止）と `on_mouse_move`（移動量参照）を連動する。カーソル禁止フラグ（`block_cursor`）は `HookSession` 生成時に移動方向ジェスチャキーの有無から事前計算し、セッション変化時に `GestureEventHandler.update()` へ通知する。
- ジェスチャ動作の送出には `output_driver.py` を使用する。
- キー・ボタンの監視には `input_driver.py` を使用する。
- `[SPEC-CONFLICT-DETECT]` `[SPEC-INVALID-UP-DETECT]` のGUIポップアップは `tkinter.messagebox` で実装する。
- `[SPEC-SELF-EVENT-FILTER]` の実現には `output_driver.py` と `input_driver.py` の両方が同一の識別子を参照する必要がある。識別子は固定定数をデフォルト値とするが、両ドライバに設定インターフェースを設け、外部から変更可能とする。
- `[SPEC-ACTION-DOWN-KEY-TRACKING]` の実現には `HookSession` がダウン発行済みジェスチャキー一覧をスタックとして管理し、セッション破棄時に逆順で走査してアップを発行する機構が必要。
- `[SPEC-MOVE-COOLDOWN]` の実現には `HookSession` が「最後に確定した方向」と「その時刻」の2値を保持するだけでよい。方向確定時に直前の方向と一致し経過時間がクールダウン未満であれば動作をスキップ、別方向であれば両値を更新して通常実行する。
